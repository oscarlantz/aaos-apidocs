WorkServiceV2 Service

Service for performing different programming jobs on keys.

Replaces the old version: v1 of work service.

Namespace
http://cliq.shared.assaabloy.com/ws/work/v2/
WSDL
workService_v2.wsdl
Path

resetKey Operation

Reset a remote enabled key, creates remote updates to reset the key.

This method is normally used in conjunction with the handInKey(String keyIdentity) method in order to ensure that the key is return without any authorisation and also inactivated.

Note, depending on the current load of the system, creation of a remote programming job might take one to several minutes.

These operations will be performed depending on key type:

  • Dynamic key: removes any access profiles and explicit authorisations, disables schedule and validity.
  • Quartz key: disables schedule and validity.
  • Normal key: disables validity.

Note, if the key is CLIQ Connect Dynamic or Quartz key and has support for PIN validation also PIN validation will be disabled.

Key system requirements:

  • Key system must be remote enabled.

Key requirements:

  • Key must have support for remote updates.

Input Parameters
name type description
keyIdentity string Identity of the key that should be reset.
Faults
name type conditions
SystemException SystemException Thrown if the system was configured incorrectly or other unexpected errors occurred.
IdentityNotFoundException IdentityNotFoundException Thrown if the given key identity could not be found.
InvalidDataException InvalidDataException Thrown if the key is a c-key or key has been deleted.

updateKeyCylinderAuthorisations Operation

Update cylinder authorisations(access rights) for a dynamic key. This operation will put "wanted" cylinder authorisations for the key and if required a remote programming job will be created.

Note, depending on the current load of the system, creation of a remote programming job might take one to several minutes.

Key system requirements:

  • Key system must be remote enabled.

Key requirements:

  • Only keys of type 'dynamic' (programmable) are valid.
  • Key must have support for remote updates.

Plug requirements:

  • Key must have mechanical and as well as electronic access in all supplied plugs
  • Plug must be operational.
  • Plug's cylinder must be installed.

Input Parameters
name type description
keyIdentity string Identity of the key that should receive the update.
cylinderPlugIdentities authorisationUpdate Authorisation update containing a list of plug ids where the key should have access. If the list of authorised cylinder plugs is empty, all existing cylinder plug authorisations on the key will be deleted.
Faults
name type conditions
InvalidDataException InvalidDataException Thrown if the input data is invalid or the list of authorised cylinder plugs contains identities that does not exist.
SystemException SystemException Thrown if the system was configured incorrectly or other unexpected errors occurred.
IdentityNotFoundException IdentityNotFoundException Thrown if the given key identity could not be found.

updateKeyValidity Operation

Update validity of a key to a new validity, the operation is accepted regardless if the newValidity period already is enabled or not.

This operation will put "wanted" newValidity on the key, already pending key newValidity updates will be replaced.

handOutKey Operation

Hand out a key to a person. For dynamic keys, this method also implies that access profiles will be copied from the person to the key.

Note, this operation is only supported for user keys.

Dates must be on the format yyyy-MM-ddTHH:MMZ (UTC time), yyyy-MM-ddTHH:MM+01:00 or yyyy-MM-ddTHH:MM+01 which is a subset of ISO 8601 (the latter two examples represent 1 hour offset from UTC). See the specification for more information.

A complete hand-out procedure does normal implies that authorisations-, validity-, PIN validation-, schedule-updates and also that audittrail are requested, se further the description of methods listed below to add any of these operation to your hand-out procedure:

  • updateKeyCylinderAuthorisations(String, AuthorisationUpdateDTO)
  • updateKeyValidity(String keyIdentity, ValidityDTO)
  • updateKeyPinValidation(String keyIdentity, PinValidationDTO)
  • updateKeySchedule(String keyIdentity, ScheduleDTO)
  • requestKeyAuditTrail(String keyIdentity)

Input Parameters
name type description
personIdentity string The identity of the person that will receive the key.
keyIdentity string The identity of the key to hand out.
handOutDate string The date when the key was handed out.
handInDate string The date when the key should be handed in, parameter is optional. Both null (no XML element in the SOAP request) and the empty string (empty XML element in the SOAP request) are accepted.
copyAccessProfiles boolean Defines that the access profile should be copied from person to key , parameter is optional and default true.
Faults
name type conditions
SystemException SystemException Thrown if the system was configured incorrectly or other unexpected errors occurred
InvalidStateException InvalidStateException If the key has already been handed out.
InvalidDataException InvalidDataException If any of the input dates are invalid or given key is a C-Key.
IdentityNotFoundException IdentityNotFoundException If either the person or the key could not be found.

declareKeyLost Operation

Declare a key as lost.

If the key was handed out before declared lost, it will no longer be considered handed out.

For remote enabled keys, all pending jobs will be cancelled except for audit trail requests and a remote job to invalidate the key will be issued.

The key will be black-listed in all affected cylinders.

This operation is only supported for user keys.

assignCylinderJobsToCKey Operation

Assigns cylinder jobs to a particular c-key.

Only the jobs that are applicable will be assigned.

Input Parameters
name type description

Faults
name type conditions
SystemException SystemException Thrown if an unrecoverable error occurs.
IdentityNotFoundException IdentityNotFoundException Thrown if the given cylinder or key identity could not be found.
InvalidDataException InvalidDataException Thrown in case of data inconsistency. Possible failure codes:
  • CKEY_REQUIRED - given key must be a c-key.
InvalidStateException InvalidStateException Thrown if current c-key or cylinder state doesn't allow to assign new jobs. Possible failure codes:
  • CKEY_BROKEN - c-key is marked as a broken and cannot be programmed.
  • CKEY_LOST - c-key is marked as a lost and cannot be programmed.
  • CKEY_BLOCKED - c-key is marked as a blocked and cannot be programmed.
  • CKEY_PROGRAMMING_NOT_ALLOWED - c-key does not have programming permission to assign job.
  • CYLINDER_BROKEN - not possible to assign jobs to the broken cylinder.
  • CYLINDER_DELETED - specified cylinder is deleted.
  • CYLINDER_JOB_ALREADY_ON_CKEY - cylinder plug is already assigned to the c-key.
  • CYLINDER_PLUG_LOCKED - one of the cylinder plug is locked for programming. Job could not be assigned to the c-key.
  • CKEY_HAS_AUTO_AUDIT_TRAIL - automatic audit trail retrieval for selected c-key is activated. It is not possible to to assign audit trail job.
  • NO_CYLINDER_JOBS_TO_ASSIGN - there are no cylinder jobs that can be assigned to the selected c-key.

This operation will put "wanted" PIN validation on the key, already pending key PIN validation updates will be replaced.

Note, depending on the current load of the system, creation of a remote programming job might take a number of minutes.

Key requirements:

  • Only keys of type 'dynamic' and 'quartz' are valid.
  • Key must have support for remote updates.

Faults
name type conditions

    • - Request data invalid

IdentityNotFoundException IdentityNotFoundException Thrown if no key with the given identity could be found.
InvalidStateException InvalidStateException Thrown in case of invalid business state for requested action.

    • - Attempt to update the Pin Validation parameters of key with scheduled disabling of Key Validity.
