CLIQ Web Manger - Web Services

The following documentation is the latest and recommended version of CWM web services. API v1 was removed in CWM 10 version.

Separate Apache server is setup for web service integration on port 8989.

The web service integration in CLIQ Web Manager publishes three interfaces for integration:
  • ImportService - For importing persons (potential key holders).
  • QueryService - For querying CLIQ Web Manager for information about CLIQ elements, audit trails etc.
  • WorkService - For performing remote operations on keys, e.g. adding cylinder plug authorisations and validity.

When CLIQ Web Manager is deployed the web service interfaces are published at the following paths within the application:

  • /ws/import/v2/?wsdl
  • /ws/query/v2/?wsdl
  • /ws/work/v2/?wsdl

example: https://localhost/CLIQWebManager/ws/work/v2/?wsdl.

The authentication between client application and CLIQ Web Manager web service is mutual (two-way), which means that it is expected that the client application verifies the identity of the server and the server verifies the identity of the client application. A specific web service client certificate is used to identify and authenticate a client application to the web service interface. The client certificate contains information that enables use of CLIQ Web Manager for a specific locking system (MKS).

auditTrailEntry An AuditTrailEntry represents one audit trail entry collected from a CLIQ element (key or cylinder). The entry has a unique identity that can be used for separating two entries from each other. The entry contains information about the element's interaction with another CLIQ element.

E.g. If an audit trail entry is read from a CLIQ key, it may contain entries created from interaction with Cylinders (access), C-keys (programming), or Remote Programming Devices (programming). If an audit trail is collected from a cylinder, it may contain entries created from interaction with Keys (access), C-keys (programming), or Local Programming Devices (programming). Other element interactions may also occur.

Audit trails come in two flavours:

  • NormalAuditTrailEntry - For interaction with elements in same MKS
  • ForeignAuditTrailEntry - For interaction with elements in foreign MKS
Normal entries contain information about the other element, while foreign audit trail entries only contain limited information needed to identify the element in the foreign system.

pagination Used to specify how much data is to be returned by the web service call. The first result parameter combined with the max results parameter allows splitting up a large number of results for iteration. This is done by asking for the first row and a max number of results (e.g. 10) and in the subsequent call asking for the 11th row and the same max amount of results and so on.

For example: pagination.firstResult=0 and pagination.maxResults=10 will give you the first 10 rows, 1-10 and pagination.firstResult=10 and pagination.maxResults=10 will give you the next coming 10 rows , 11-20

revalidation Represents a revalidation configuration for a CLIQ key.

Only keys of type 'dynamic' and 'quartz' are applicable and valid.

A revalidation configuration consists of two parts: - one time update - interval in minutes

The one time update is a flag which, when set, allows for one revalidation only whereby no more validity updates are issued. The interval in minutes specifies the validity extension issued to the key each time it is inserted into a remote PD.

schedule Schedule represents a CLIQ Key schedule.

Depending on configuration, a key supports a number of time period slots. Each slot consists of a time interval (from week day and time, to week day and time) when the key can be granted access. If no time periods are set, the key's existing time periods will be deleted.

Only keys of type 'dynamic' and 'quartz' are applicable and valid.

A fault is returned if the maximum number of slots is exceeded.

timePeriod TimePeriod represents a schedule time interval slot. A time period contains a start date and and a start time, a stop date and a stop time. A period can never cross midnight between Saturday and Sunday.

Optionally a cylinder plug identity can be supplied to define a time period slot only for a specific cylinder plug. If no cylinder plug identity is supplied the schedule applies to all cylinder plugs where the key has access.

A fault is returned if periods crosses midnight between Saturday and Sunday.

validity Class representing a CLIQ key validity. This is used to define the time period when the key is valid for access.

Key validity can be defined as:

  • ALWAYS
  • NEVER
  • DATES
If defined as DATES, at least 'from' must be supplied. Leaving out the 'to' field creates a open-end validity.

Optionally a revalidation configuration can be supplied.
